## Governance Policy

<-- Governance Policy — Claude Code - Agentic EDI Development -->


**Scope:** This policy applies to all developers, contractors, and AI agents using Claude Code to:Generate EDI format specifications (e.g., ANSI X12, EDIFACT, XML).Create mapping logic between EDI structures and internal database systems/ERPs.Automate testing and validation of EDI documents.Refactor or generate code for EDI parser microservices. Excludes: This policy does not apply to non-production, sandboxed testing of Claude Code, provided no real production EDI data or sensitive customer metadata is used.

**Autonomy boundaries:** Sandboxed Execution: Claude Code must run within a containerized environment (e.g., Docker) or a sandboxed virtual machine. It must not have direct shell access to production servers. — never auto. ReadOnly Access to Production: Claude Code is restricted to ReadOnly access to production databases or message queues when analyzing existing EDI mappings. — human approval required. Approval-Required Actions: Any git commit, git push, or direct execution of kubectl / terraform commands generated by Claude Code requires manual human review and approval within the terminal or through GitOps workflow. — human approval required. Secret Management: Claude Code is prohibited from accessing plain-text environment variables or secrets. It must interface only through approved Secret Managers — human approval required. Model Restrictions: Only approved models (e.g., Claude 3.5 Sonnet or newer) with enabled Zero Data Retention (ZDR) are allowed for handling sensitive PII/PHI in EDI data. — never auto.

**Escalation triggers:** Trigger 1: Permission Denial: Attempting to access files outside the \./src/edi/ project directory or executing unapproved system-level commands. Trigger 2: High-Risk Code Generation: Generation of code that could modify production data, open network sockets, or change authentication mechanisms. Trigger 3: Cost/Token Threshold: Exceeding an API spend limit of \(\$50\) USD within a 4-hour period (suggested limit, adjust based on scale). Trigger 4: Security/Safety Violation: Flagging of content by the Constitutional AI safety layer. Trigger 5: Failure to Compile/Test: Failing two consecutive attempts to generate valid, compiling EDI code.

**Audit cadence:** Real-time — Automated Logging: Claude Code CLI traffic must be routed through an AI Gateway (e.g., Kong AI Gateway) to log all prompts, responses, tool calls, and file modifications. (Integration Solution Architect). Weekly — Peer Review Audit: A randomly selected sample of 10% of Claude-generated EDI mappings must be reviewed by a human senior engineer. (Senior Integration Engineer). Quarterly — Compliance Review: A full audit of the governance policy, checking for compliance with the 2026 EU AI Act regarding transparency and human oversight. (Product Owner).

**Regulatory exposure (EU AI Act / other):** Transparency (EU AI Act Article 50): EDI documentation must explicitly state that the mapping was generated with the assistance of Claude Code.
Data Protection (GDPR): If EDI data contains EU personal data, ZDR must be enabled to prevent data retention by Anthropic.. Risk tier: limited. Controls: Internal Controls: Organizations must maintain a Governance Log proving that humans supervised the final deployment of EDI systems, even if developed by agents..
